# 1号文档：AIChan 设计文档

## 1.1 设计目标
AIChan 的核心目标不是“让所有输入尽快触发模型”，而是建立一个可控、可解释、可扩展的认知循环。
系统采用统一感知入口、统一调度中枢、统一推理核心的设计，确保行为稳定且可观测。

## 1.2 核心设计理念（The Philosophy）
AIChan 的架构理念以两条主线为核心：
- 控制反转的拉取模式（Pull-based）
- 大一统的多模态工具（Fat Tool）

这意味着系统不再以“事件来了就立即推给 LLM”的直连模式运行，而是让编排层按可控节奏消费任务；
同时将外设能力抽象为统一可调用能力，降低扩展成本和模型理解成本。

### 1.2.1 万物皆消息，万物皆工具
在 AIChan 的世界观里，外设既是输入感官也是执行肢体：
- 作为感官：监听物理世界并转化为标准信号/消息。
- 作为能力：向系统暴露统一控制接口，由编排层或 LLM 在边界内调用。

### 1.2.2 Pull 优于 Push（控制反转）
AIChan 将外部刺激先进入统一调度路径，再由推理 Agent 按节奏拉取处理。
这样做的核心收益是：
- 通过背压（Backpressure）吸收突发并发。
- 避免高峰期“消息洪峰直接打爆 LLM 调用链”。
- 保证处理顺序稳定与状态可追踪。

## 1.3 全面 Pull（拉取）模式与单核大脑

### 1.3.1 设计定义
- 全面 Pull：外部刺激只能作为信号进入统一调度路径，不能绕过编排层直接触发 LLM。
- 单核推理 Agent：`agent.Agent` 作为唯一 LLM 推理入口，以单线程语义顺序处理上下文。

### 1.3.2 设计原则
- 调度层负责“何时处理”，`agent.Agent` 负责“如何思考”。
- 输入侧是生产者，推理侧是消费者，边界清晰。
- 任意时刻只允许一个主推理上下文处于执行态。

### 1.3.3 设计收益
- 并发可控：避免多输入并发直冲 LLM 导致上下文竞争。
- 顺序稳定：拉取与队列语义天然提供时序一致性。
- 可扩展：新增输入源只需遵守信号与通道协议。

## 1.4 拟人化注意力机制与惰性加载

### 1.4.1 核心认知假设
AIChan 采用“单焦点注意力”模型：
- 大脑每个时刻只聚焦一个刺激。
- 其他刺激进入等待路径，等待注意力释放。

### 1.4.2 注意力循环
1. 感知：交互层或通道插件采集外界刺激并形成标准消息。
2. 发信号：运行层发出信号，告知“该通道出现新刺激”。
3. 聚焦：`SignalProcessor` 拉取本轮增量消息并交给 `agent.Agent`。
4. 推理：`agent.Agent` 执行 LLM 推理与工具调用。
5. 释放：`SignalProcessor` 将结果写回通道，进入下一条信号。

### 1.4.3 惰性加载意图
在多通道高频场景中，优先进行低成本缓存与聚合，再批量拉取上下文进入推理，
以降低 Token 消耗并提升交互流畅性。

## 1.5 系统器官化分层（Conceptual Organs）
- `core`（基因组）：定义全局数据结构、抽象契约与基础能力。
- `plugins`（感官与外设）：与物理世界交互，负责采集与执行。
- `hub`（中央神经枢纽）：承载异步信号缓冲与心跳调度（可扩展中枢）。
- `agent`（推理层）：执行推理与工具编排。
- `memory`（海马体）：负责短期/长期记忆的检索与沉淀。

## 1.6 关键设计模式（Key Patterns）

### 1.6.1 生产者-消费者模式
- 生产者：交互循环/外设监听，将刺激写入统一信号路径。
- 消费者：编排层按节奏消费，驱动 `agent.Agent` 串行推理。

### 1.6.2 多模态胖工具模式（Fat Tool）
通过结构化参数（如 `action`）把同领域能力收敛到稳定接口，降低工具数量爆炸导致的注意力分散。

> 当前实现补充：系统在理念上保留 Fat Tool 方向，同时已落地“工具插件/通道插件”分型，
> 以确保通道外设不会被 LLM 直接越权操作。

## 1.7 当前实现映射（补充整理）
- 默认运行模式为双进程协作：`main.py` 启动 AIChan 核心并内嵌 `cli_server`，`cli_client.py` 独立运行。
- 交互输入先由 `cli_client` 通过 HTTP 写入 `cli_server`，服务端仅更新未读状态，不主动推送信号。
- AIChan 侧由 `CLIChannelPlugin` 每秒轮询未读状态，发现 AI 侧有未读后触发 `AgentSignal(channel=...)` 入 `SignalHub` 队列。
- `SignalHub` 心跳线程串行消费信号并调度 `SignalProcessor.process_signal(...)`。
- `SignalProcessor` 按通道维护已处理游标，仅消费由 `CLIChannelPlugin`（HTTP 客户端）拉取到的增量 user 消息。
- `agent.Agent` 输出最终文本，`SignalProcessor` 将其写回通道（`role=assistant`）。
- `cli_server` 承担消息存储与收发管理，AIChan 与用户端都以 client 身份访问该服务。
- 仅工具插件绑定给 LLM，通道插件由编排层调用。
- `SignalHub` 作为统一调度入口，已具备多入口扩展基础。

## 1.8 架构演进收益（Benefits）
- 极强的并发吸收能力：核心推理频率由系统自身节奏控制。
- 高解耦扩展：新增平台时优先在 `plugins` 边界内扩展，不侵入推理主链路。
- LLM 友好：结构化能力暴露降低上下文冗余与调用歧义。

## 1.9 设计约束
- `agent.Agent` 是唯一 LLM 调用入口。
- 通道读写由编排层驱动，交互层只负责输入与展示。
- 插件扩展必须明确归属“工具”或“通道”，避免混型。
- 当启用多入口并发接入时，建议统一通过 `SignalHub` 入队调度。
